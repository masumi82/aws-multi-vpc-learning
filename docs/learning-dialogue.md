# 対話で振り返る AWS マルチ VPC 構築

> 2026-05-13 に実施した個人学習プロジェクト「aws_sekei」を、
> 生徒と先生の対話で振り返ります。

---

## 第1幕: なぜこの構成?

**生徒**: 先生、ぼく AWS の練習で「VPC を 2 つ、AZ を 3 つ、ALB をつけて、Aurora をマルチ AZ で」って構成を作ってみたいんですけど、これって何がうれしいんですか?

**先生**: いい問いから入りましたね。一つずつ意味があります。
- **3 AZ** にするのは、AWS の SLA 上 Aurora のフェイルオーバーやマルチ AZ 冗長化を「ちゃんと体験する」ための最小単位です。2 AZ だと "高可用性"の練習にはなりますが、3 AZ にすると **Quorum (多数決ベース)** で動く Aurora の本来の仕組みが見えます。
- **VPC を 2 つ** にするのは、ネットワーク分離 (環境分離・テナント分離・役割分離) の **本物の練習**になるからです。

**生徒**: なるほど。でも最初、先生は「VPC-A は Web 層、VPC-B は App+DB 層」って提案してくれましたよね。途中で「Prod / Dev 分離」に変えました。なぜですか?

**先生**: あなたが「UI は S3 から配信したい」と言ったからです。覚えていますか?

**生徒**: あ、はい。ガントチャートみたいなシンプル SPA だから、EC2 で Nginx 立てて配信するのは大げさかなって。

**先生**: その判断は正解でした。S3 + CloudFront は静的 SPA 配信の **業界標準**です。コストは EC2 の 1/30 以下、運用負荷ほぼゼロ、CDN で速い。
ただし — **S3 は VPC の外側のマネージドサービス**なので、当初設計の「Web 層 VPC」が役目を失ったんです。「2 VPC」の前提だけ残して、役割を **Prod / Dev 分離** に組み替えたわけです。

**生徒**: あー、要件 (VPC 2 つ) は維持しつつ、内部の役割を最適化したんですね。

**先生**: その通り。要件と実装は分けて考える、という設計の基本動作です。

---

## 第2幕: なぜ ECS Fargate?

**生徒**: App 層は最初「EC2 + Auto Scaling Group」だったのが「ECS Fargate」に変わりました。これも何で?

**先生**: あなたが「シンプルな CRUD API」と教えてくれたからです。CRUD API は **サーバーレスでもコンテナでも軽量** に動きます。EC2 で OS パッチを当て続けるよりも、Fargate でコンテナだけ管理した方が学習価値が高い。

**生徒**: ECS と EC2 + ASG って、Terraform で書くと何が違うんですか?

**先生**: 大きな違いは 3 つ。
1. **`target_type = "ip"`** が必須 — Fargate の awsvpc ネットワークモードでは ENI 単位で IP が振られるので、ALB Target Group は instance ベースでは指せません
2. **`AmazonECSTaskExecutionRolePolicy`** という Managed Policy がある — ECR pull / awslogs / Secrets Manager 参照に必要な権限がほぼ網羅されている
3. **タスクごとに 2 つの IAM Role** — `ExecutionRole` (起動側) と `TaskRole` (アプリ自身) を使い分ける

**生徒**: そういえば `tests/integration/run.sh` の I10 で「TG target_type=ip」を検証してましたね。

**先生**: そうです。テストで明示的に検証しているのは、**Fargate に EC2 用の TG をつないでハマるパターンを最初から防ぐ**ためです。

---

## 第3幕: なぜ Aurora?

**生徒**: Aurora じゃなくて普通の RDS PostgreSQL でも良かったですよね?

**先生**: 良かったです。ただし違いは「3 AZ にまたがる」を **本当の意味で体験できる** かどうか。
- RDS Multi-AZ: スタンバイ 1 台が別 AZ にあるだけで、普段は使われない
- **Aurora**: Writer 1 + Reader 複数を **別 AZ に配置**、Reader は **読み専用エンドポイント経由で実際にトラフィックを受ける**

要件で「マルチ AZ にまたがる」を体験したいなら、Aurora の方が **学習が濃い**です。

**生徒**: パスワード管理が `manage_master_user_password = true` ってなってましたが、これって珍しい設定ですか?

**先生**: 比較的新しい機能で、**Aurora 側が Secrets Manager にシークレットを自動生成・自動ローテーション** してくれます。これを使わない場合、自分で `random_password` を作って Secrets Manager に PUT する手間が発生します。学習用にはこちらで充分。実務でも増えてきている設定です。

**生徒**: ECS のタスク定義で `secrets` ブロックで `${secret_arn}:username::` みたいに書いてましたね。

**先生**: そう、JSON のキー単位で取り出せる構文です。`${ARN}:JSON_KEY::` で `username` と `password` を別々の環境変数として注入できる。これも実務で頻繁に使うパターンです。

## 第4幕: なぜ Terraform?

**生徒**: 「Terraform / CDK / CloudFormation のどれにしますか」と聞かれて Terraform を選びましたが、本当にこれで良かったんでしょうか?

**先生**: 学習目的なら **Terraform が最も汎用的**です。理由は 3 つ:
1. **業界での採用率が圧倒的** — 求人検索すると分かります
2. **AWS 以外にも転用できる** — GCP / Azure / Cloudflare / Datadog などにも同じ HCL 構文
3. **State という概念が明示的** — クラウドリソースの「現状」を Terraform がどう管理しているかを意識するクセがつく

CDK は AWS への深い学習には向いていますが、**抽象が高すぎて中で何が起きているか見えにくい**。CloudFormation は YAML が冗長で読みにくい。学習の最初は Terraform が最適です。

**生徒**: ローカル State にしたのもなぜですか? S3 backend が実務的って言ってましたよね?

**先生**: 「検証完了後すぐ削除」が前提だからです。S3 backend を作ると、その S3 と DynamoDB 自体も Terraform 管理外で別途準備する必要があり、削除も別途。**学習中に集中したいことが「マルチ VPC + Aurora + ECS の構築」**なので、State 管理は最小化しました。

**生徒**: 実務に行ったら S3 + DynamoDB backend にすべき?

**先生**: チーム開発なら必須です。State はチーム共有資産で、ローカル PC に置くと「他人の apply 結果が反映されない」「state ファイルが破損したら詰む」「並行 apply で競合する」などの事故が起きます。

## 第5幕: なぜテストを 3 層に?

**生徒**: 先生、テストを Static / Integration / E2E の 3 層に分けましたよね。普通のテストってこんなにレイヤー分けないと思うんですが…

**先生**: 普通のアプリ開発 (Unit / Integration / E2E) の **インフラ版**だと思ってください。それぞれ役割が違います。

| Layer | 検証対象 | コスト | タイミング |
|---|---|---|---|
| Static | コード自身 (構文・規約) | 0 | コミット前・毎回 |
| Integration | AWS リソースが定義通りか | 課金中 | apply 後 |
| E2E | エンドユーザー体験 | 課金中 + ブラウザ | apply 後・Deployed 後 |

**生徒**: 今回 Static は 5/5 PASS、Integration は 24/24 PASS、E2E は 6/7 PASS でしたね。

**先生**: そう、そして **最も価値が高かったのは E2E の "FAIL" 1 件**です。

---

## 第6幕: 一番の発見 — CloudFront SPA+API の落とし穴

**生徒**: E2 の「`/api/` で Nginx の welcome ページが返ってこなかった」やつですね。

**先生**: あれを覚えていますか? Playwright が「`/api/` で 200 が返ったけど、中身が S3 の index.html だった」と FAIL を出した瞬間です。あなたはどう思いましたか?

**生徒**: 最初「テストが間違ってるのかな」って思いました。でも curl で確認したら、ヘッダに `x-cache: Error from cloudfront` と `server: AmazonS3` が出ていて、本当に S3 から返ってきてました。

**先生**: そこが学びの核心です。**テストが正しく、設計が想定通りに動いていなかった**。
何が起きていたかというと:

```
1. ブラウザ → CloudFront `/api/` → ALB → Nginx
2. Nginx は `/api/` というパスを知らない (デフォルトでは `/` しかルートにない)
3. Nginx が 404 を返す
4. CloudFront の custom_error_response 404→/index.html が発火
5. クライアントは S3 の index.html を 200 で受け取る
```

**生徒**: つまり Nginx は 404 を返したのに、CloudFront がそれを「404 → 200 (index.html)」に書き換えちゃったってことですか?

**先生**: そうです。これは **CloudFront の `custom_error_response` が Behavior 単位ではなく Distribution 単位で適用される** という仕様による落とし穴です。

**生徒**: でも私たち、SPA のフォールバック (存在しないパスは index.html を返す) は意図して入れましたよね?

**先生**: そこがジレンマです。
- **SPA のため**: 存在しないパスは index.html 200 を返したい (E3 で検証成功)
- **API のため**: エラーは 4xx/5xx でそのまま返したい

この 2 つが Distribution 単位の設定で衝突します。

**生徒**: 修正するにはどうすれば?

**先生**: 4 つくらい選択肢があって、それぞれ学習コストが違います。

| 方針 | 概要 | 難易度 |
|---|---|---|
| CloudFront Function | viewer-response で `/api/*` だけ error 書き換えを抑制 | ★★★ |
| Distribution 分離 | UI と API で別 Distribution | ★★ |
| Nginx 側で対応 | `/api/` でも 200 を返す nginx config | ★ |
| 本物の API へ差し替え | dummy ではなく FastAPI 等で適切なエラーを返す | (本来の運用) |

**生徒**: 本回はどれもやらなかったんですね。

**先生**: そう、**学習の本筋 (マルチ VPC 構築 + テスト体験) から外れる**からです。**「テストが課題を発見した」という事実だけ持ち帰る**のが今回のゴールでした。

**生徒**: なるほど…テストの価値って「PASS させること」じゃなくて「実態を可視化すること」なんですね。

**先生**: その理解、すごく大事です。テストが PASS したら「設計通り」、FAIL したら「設計と実態にズレがある」。どちらも価値がある情報です。

## 第7幕: コストとライフサイクル

**生徒**: 1 時間で数十円〜100 円くらいでしたね。安いって思いましたけど、これって油断ですか?

**先生**: 油断です。今回は **destroy を忘れなかった**から数十円で済んだ。もし destroy を忘れて 1 日放置していたら **$22 (約 3500 円)**、1 週間で **$150 (約 24,000 円)** です。

**生徒**: NAT Gateway と EIP が特に気をつけるべきって何度も言ってましたね。

**先生**: そう、**時間課金のリソースは止められない**。停止という概念がなくて、削除しかない。EC2 なら停止できますが、NAT Gateway は削除しないと課金が続きます。EIP も「アタッチされていない時間」に課金が発生する仕様で、忘れがちなトラップです。

**生徒**: `cleanup-checklist.md` ってあれだけ細かく作ったの、過剰かと思いましたけど…

**先生**: 過剰じゃありません。**自分のお財布を守る装備**だと思ってください。実務でも、開発環境を delete し忘れて月末に部長に怒られる若手、本当によく見ます。

## 第8幕: なぜ Playwright MCP でエビデンス?

**生徒**: ブラウザの試験は Playwright MCP でやりましたが、これって何のためにエビデンスを残すんですか?

**先生**: 3 つ意味があります。
1. **再現性** — 「あの時こう動いた」を後日確認できる (スクショ・ネットワークログ)
2. **監査・引き継ぎ** — 自分が手放した後も別の人が状況を理解できる
3. **比較** — 設計を変更したあとに「以前と同じ挙動か / 変わったか」を客観的に判定できる

**生徒**: 今回 `tests/evidence/2026-05-13/` に PNG とか YAML とかいろいろ保存しましたね。

**先生**: そう、**スクショだけでなく、アクセシビリティスナップショット (DOM 構造) とネットワークログも**残しました。スクショは見た目、スナップショットは構造、ネットワークログは事実 (status code とか) を記録します。

**生徒**: E2 のスクショ、`/api/` を叩いてるのに UI が表示されてるのが衝撃的でした。

**先生**: あれが**ビジュアル証跡**の威力です。文章で説明されても「ふーん」だけど、スクショで見ると「ほんとに UI じゃん!」って驚く。後日この設計を直すときに「直す前はこうだった」という強い動機になります。

## 第9幕: 振り返り

**生徒**: 今回の学習で、自分の中で一番大きく変わったことってなんでしょう。

**先生**: 自問してみてください。プロジェクト前と後で、何が変わりましたか?

**生徒**: …うーん。
- AWS のリソースが頭の中で「点」じゃなくて「線でつながった」感じがします。VPC があって、Subnet があって、SG があって、ALB があって、ECS があって、それが Aurora と話して…って実体験できた
- Terraform で 60 リソース書いて apply して destroy したのは、AWS マネジメントコンソールでポチポチやってた時とは全然違う「再現可能性」を感じました
- テストが**設計の盲点を発見した**のは正直びっくりしました。テストってバグ検出だけじゃないんですね

**先生**: 全部 大事な気づきです。特に最後の「テストが設計の盲点を発見」は、**設計者とテスト設計者を兼ねる時のジレンマを乗り越える経験**でした。自分で書いた設計の弱点を、自分で書いたテストが暴く。これはペアプロでもなかなか起きない、ひとり完結ならではの体験です。

**生徒**: 次は何を学べばいいでしょうか?

**先生**: 優先度順にいくつか。

1. **本物の CRUD API を ECR にデプロイ** — 今回 dummy nginx だったので、FastAPI / Express で本物の `/api/users` 系を作る。E2 課題と合わせて解決できる
2. **CloudFront Function** で SPA fallback を `/api/*` から除外する設定を書いてみる
3. **CI/CD** (GitHub Actions で `terraform plan` を PR に自動コメント)
4. **CloudWatch Alarm** + SNS でアプリエラー検知
5. **prod 環境** にも apply して、2 環境同時運用 (今回 dev のみ)

**生徒**: 1 個ずつ消化していきます。先生、ありがとうございました。

**先生**: お疲れさまでした。**設計 → 実装 → テスト → 撤収** までを 1 サイクル回せた経験は、コードリポジトリと一緒に手元に残ります。後日この `docs/` と `tests/` を読み返したら、必ず今日の感覚を思い出せますよ。

---

## まとめ: 今回学んだ 5 つのコア

1. **要件と実装は別物**: 「VPC 2 つ」という要件を維持しつつ、内部の役割は柔軟に組み替えて良い (Web/App → Prod/Dev)
2. **マネージドサービスは VPC の外**: S3 / CloudFront を採用すると VPC 設計が変わる
3. **CloudFront の `custom_error_response` は Distribution 単位**: SPA + API 混在の落とし穴
4. **テストの価値は PASS させることではなく、実態を可視化すること**: E2 の FAIL は成果
5. **時間課金リソースは「停止」ではなく「削除」**: NAT / ALB / Aurora / CloudFront / EIP を意識せよ

すべて `docs/architecture.md` / `docs/final-report.md` / `tests/evidence/2026-05-13/` と現物のコードに残っています。
